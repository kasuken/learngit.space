Embed team standards directly in the repo

Mission Briefing

Every space mission operates according to specific protocols, safety guidelines, and communication standards. These aren't just suggestions—they're mission-critical rules that ensure consistency, safety, and success across all operations.

Similarly, every development team has its own "Git culture"—specific conventions for commit messages, branch naming, pull request formats, and code review processes. The challenge is ensuring that these standards are consistently applied, especially when using AI assistance.

GitHub Copilot can learn and follow your team's specific conventions when you provide clear, repository-specific instructions. This ensures that AI suggestions align with your team's established practices rather than generic conventions.

Mission Objectives

  • Create comprehensive `.github/copilot-instructions.md` files
  • Define team-specific Git conventions and standards
  • Validate Copilot adherence to custom instructions
  • Establish repository-wide consistency for all contributors

Hands-On Lab: Custom Copilot Instructions

Step 1: Creating the Instructions File

Let's create a comprehensive instruction file for your repository:

Create: .github/copilot-instructions.md 
# GitHub Copilot Instructions for [Your Project Name]

## Git Workflow Standards

### Commit Message Format
- **MUST** follow Conventional Commits specification
- **Format**: `type(scope): description`
- **Types**: feat, fix, docs, style, refactor, test, chore
- **Max length**: 50 characters for subject line
- **Include ticket number**: Format as `[JIRA-123]` at the end

**Example**: `feat(auth): add OAuth login [JIRA-456]`

### Branch Naming Convention
- **Feature branches**: `feature/JIRA-123-short-description`
- **Bug fixes**: `fix/JIRA-456-bug-description`
- **Hotfixes**: `hotfix/critical-issue-description`
- **Releases**: `release/v1.2.3`

### Pull Request Standards
- **Title**: Must match commit message format
- **Description**: Include:
  - What changed and why
  - Testing instructions
  - Screenshots for UI changes
  - Breaking change notices

## Code Review Process
- All PRs require at least 2 approvals
- Include `@team/reviewers` for automatic assignment
- Use `WIP:` prefix for work-in-progress PRs
- Link related issues using "Closes #123" syntax

## Release Management
- Use semantic versioning (SemVer)
- Tag releases with format `v1.2.3`
- Generate changelog from commit messages
- Include migration instructions for breaking changes

Step 2: Advanced Team-Specific Rules

Let's add more detailed conventions specific to your team's workflow:

Advanced Instructions 
## Team-Specific Conventions

### Commit Scope Guidelines
- `api`: Backend API changes
- `ui`: Frontend user interface
- `db`: Database schema or migration
- `auth`: Authentication/authorization
- `config`: Configuration changes
- `ci`: Continuous integration changes

### Required Commit Message Elements
1. Type and scope (mandatory)
2. Brief description (mandatory)
3. Body with detailed explanation (for complex changes)
4. Footer with issue references (mandatory)

### Forbidden Practices
- ❌ Generic messages like "fix bug" or "update code"
- ❌ Committing directly to main/master branch
- ❌ Force pushing to shared branches
- ❌ Large commits that mix multiple concerns

### Preferred Practices
- ✅ Small, atomic commits that do one thing well
- ✅ Descriptive commit messages that explain the "why"
- ✅ Regular rebasing to keep history clean
- ✅ Squashing related commits before merging

### Integration Guidelines
- Always create PR from feature branch
- Include automated tests for new features
- Update documentation for API changes
- Verify CI/CD pipeline passes before merging

Step 3: Copilot-Specific Guidelines

Add instructions specifically for AI-assisted development:

AI Assistance Guidelines 
## GitHub Copilot Usage Guidelines

### When Generating Commit Messages
- Always include appropriate conventional commit type
- Reference the ticket number in format [JIRA-123]
- Focus on business impact rather than technical details
- Keep subject line under 50 characters
- Add detailed body for non-trivial changes

### When Suggesting Branch Names
- Use kebab-case formatting
- Include ticket number after prefix
- Keep descriptions concise but meaningful
- Follow team naming patterns

### When Creating Pull Requests
- Generate title from commit message
- Include testing checklist in description
- Add appropriate labels automatically
- Reference related issues and dependencies

### Quality Standards
- All AI-generated content should be reviewed by human
- Verify suggestions match team conventions
- Customize generic suggestions to fit project context
- Maintain consistency with existing codebase patterns

### Examples of Good AI Usage
```
Prompt: "Generate commit message for authentication changes"
Good Output: "feat(auth): implement OAuth2 integration [JIRA-123]"

Prompt: "Create branch name for fixing mobile layout issues"
Good Output: "fix/JIRA-456-mobile-layout-responsive"
```

Personal VS Code Configuration 
{
  "github.copilot.chat.commitMessageGeneration.instructions": [
    { "text": "Use conventional commit format: type(scope): description" },
    { "text": "Use imperative mood: 'Add feature' not 'Added feature'" },
    { "text": "Keep subject line under 50 characters" },

    { "text": "Include scope when relevant (e.g., api, ui, auth)" },

  ]
}
